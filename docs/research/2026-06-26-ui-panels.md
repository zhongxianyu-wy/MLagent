## 二、重点深度（每面板 top 1-2）

### 面板 A 重点 1：marimo（reactive 重渲染思想）
仓库：https://github.com/marimo-team/marimo 。把 notebook 表示成 **dataflow graph**（https://marimo.io/blog/dataflow ）：改一个 cell，静态分析出依赖它的所有 cell，自动重算。HN 团队："reactive runtime based on **static analysis**, so it's performant"（https://news.ycombinator.com/item?id=38971966 ）。还支持 lazy 模式：标记 stale 而不立即重算（适合昂贵计算）。
- **借鉴**：MLagent 改脚本时，FastAPI 重读 manifest.yaml + 对应代码段 → 推前端局部重渲染——本质是 marimo 的"输入变→依赖输出重算"，但你的依赖图是 manifest 定义的（语义步骤），比 marimo 的代码静态分析简单得多。
- **不借鉴**：marimo 是完整 notebook runtime（Pyodide/WASM），太重，不能当依赖。

### 面板 A 重点 2：Quarto Code Annotation（步骤化代码范式）
文档：https://quarto.org/docs/authoring/code-annotation.html 。代码块里给特定行打编号注释（`<1>`），右侧/下方列解释。发布博客：https://opensource.posit.co/blog/2023-03-13_code-annotation/ 。这正是 MLagent 面板 A 的形态——**代码段 + 行号高亮 + purpose 注释**。manifest 每个 step 的 key_params/purpose 对应 annotation。**仅参考 UI 范式**（Quarto 是文档生成器，不能当运行时）。

### 面板 B 重点 1：logdy-core（架构参考）
仓库：https://github.com/logdyhq/logdy-core 。单二进制、零依赖、捕获 stdout 或 tail 文件 → 内嵌 web UI 服务在本地端口。像 grep/awk 加到 PATH 即用。HN：https://news.ycombinator.com/item?id=39277079 。
- **借鉴**："自托管单进程 + 内嵌 web UI"哲学与 MLagent 完全一致（本地单用户、SessionStart 拉起）。高频日志流不卡顿的实践（博客 https://logdy.dev/blog/post/from-0-to-500-stars-logdy-github-journey-in-3-months ）。
- 对应实现：FastAPI + `watchfiles`/`asyncio` tail `log.txt` → SSE/WebSocket 推前端。

### 面板 B 重点 2：React Virtuoso（流式贴底，若用 React）
站点：https://virtuoso.dev/ ，NPM https://www.npmjs.com/package/react-virtuoso 。虚拟列表 + 一等公民 `followOutput`（贴底）/ 反转列表，专为 chat/log 流设计。
- **用点 + 坑**：`followOutput` prop（设 `"smooth"`/`"auto"`）+ `initialTopMostItemIndex` 定位最新行。
- **已知坑（必读）**：issue #317（https://github.com/petyosi/react-virtuoso/issues/317 ）——日志追加比上一帧滚动动画还快时 followOutput 失效停止贴底，需 debounce 批量更新；issue #423（https://github.com/petyosi/react-virtuoso/issues/423 ）——高频更新+缩放导致贴底闪烁；Troubleshooting（https://virtuoso.dev/react-virtuoso/troubleshooting/ ）——"列表滚不到底"通常是内层 DOM 自带 height 打断 follow。
- **高频日志务必 batch（~16ms 聚合）后再 setState**，否则掉帧。
- 若选无构建（HTMX），自己写 chunked SSE + 简单 ring buffer。

### 面板 C 重点 1：MLflow compare view + Aim Explorers（UI 设计）
- **MLflow**（https://mlflow.org/docs/latest/ml/tracking/ ｜ Databricks https://docs.databricks.com/aws/en/mlflow/visualize-runs ）：选多 run → Compare → toggle **Table View**（每行一个 run，列=params+metrics）或 **Chart View**（metric-over-time 折线）。
- **Aim**（https://aimstack.readthedocs.io/en/latest/ui/pages/explorers.html ｜ Runs Management https://aimstack.readthedocs.io/en/latest/ui/pages/run_management.html ｜ GitHub https://github.com/aimhubio/aim ）：runs 表 + Advanced Search（按 metric 值查询）+ 单 run Metrics tab 支持 **pin 常用 metric**。
- **借鉴**：MLflow Table View 列布局（run_id | 关键 param | 关键 metric）直接套 metrics.json；Aim 的 pin 设计（metric 多时 pin accuracy/loss 很实用）；chart toggle（MVP 先纯表格，后续加 metric-over-step 折线，Chart.js CDN 即可）。
- **仅参考 UI 设计，不引入依赖**。

## 三、复用结论

### 前端实现路径推荐（关键决策：要不要构建步骤）

**强烈推荐：FastAPI + Jinja2 + HTMX + Shoelace (CDN) + Alpine.js —— 无 Node 构建步骤**
- Blake Crosley 指南 https://blakecrosley.com/guides/fastapi-htmx 完整演示 FastAPI + HTMX + Alpine + Jinja2 + 纯 CSS 无构建全栈。
- Shoelace（https://shoelace.style/ ）是 framework-agnostic web components，CDN 引入，HTMX 可直接驱动（https://binaryigor.com/htmx-with-shoelace-framework-agnostic-components-in-an-example-app.html ）。
- 完全契合：本地单用户、少依赖、无 Node 工具链。

**备选（更交互富）：FastAPI + React（Vite）单页**
- fastapi-sse（https://github.com/AndreyKlychnikov/fastapi-sse ）做事件流，React Virtuoso 做日志虚拟列表。引入 Node/Vite 构建链，违背"少 Node 工具链"倾向。**仅当面板交互复杂度确实需要时才选**。

### 各面板数据流（统一：SSE 推、局部重渲染）
```
watchfiles 监听文件 → FastAPI 端点（EventSourceResponse）→ SSE text/event-stream → 前端 EventSource → 局部重渲染
```
- **面板 A**：watchfiles 监听 `<script>.py` + `manifest.yaml` → FastAPI 重读 manifest + 抽代码段（按行号切片）→ SSE 推 → HTMX `hx-swap` 替换步骤 DOM。
- **面板 B**：`asyncio` 异步 tail `runs/<id>/log.txt`（或 watchfiles 监听追加）→ SSE 流式推每行 → 前端 ring buffer（最近 N 行）+ 错误行 `class="error"` 高亮 + "暂停/继续贴底"开关。
- **面板 C**：watchfiles 监听 `metrics.json` → FastAPI 解析 → SSE 推 → HTMX 重渲染表格行；跨 run 对比时读多个 run 的 metrics.json 拼表。

**SSE vs WebSocket**：全是单向（服务端→客户端），**SSE 更合适**——自动重连、走 HTTP、`sse-starlette` 一行接入。WebSocket 仅双向需要时考虑，但写回用普通 PUT 即可，**不需要 WebSocket**。

### 人工干预编辑写回的落地
- **编辑器**：**CodeMirror 6**（https://discuss.codemirror.net/t/wanting-to-save-a-file/734 ）。~50kB tree-shaken，远轻于 Monaco（1-5MB）；Sourcegraph 从 Monaco 迁 CM6 博客（https://sourcegraph.com/blog/migrating-monaco-codemirror ）证明大规模可行；Monaco 不符合轻量约束。
  - 注意：CM6 是 ESM，无构建步骤下需用预构建 bundle 或 CodeMirror 5（更易 CDN 引入但功能弱）。
- **写回**：编辑器 onBlur/Ctrl+S → `fetch('/api/script/{id}', {method:'PUT', body: code})` → FastAPI 写回文件 + 打 `human_interventions` 标记。Monaco/CM 都不处理文件 I/O，需自己实现（issue #356 https://github.com/microsoft/monaco-editor/issues/356 ）。
- 参考：JupyterLab 编辑 cell 保存的回写机制。

### 整体可借鉴的 dashboard 骨架
**最贴合：FastAPI + HTMX + Shoelace 无构建骨架**（Blake Crosley 指南 + binaryigor 文章）。三面板用 Shoelace 的 layout 组件（tab/drawer/details）拼，每个面板一个 HTMX 驱动的局部更新区，SSE 统一推送。最符合"本地单用户、少依赖、无 Node 工具链"约束。
**次选（结构完整但更重）**：fastapi/full-stack-fastapi-template（https://github.com/fastapi/full-stack-fastapi-template ）——官方全栈 React 模板，含 Docker/认证/SQLModel，对 MVP 过重，仅看项目组织结构。

## 四、关键风险与提醒

1. **CodeMirror 6 + 无构建步骤有摩擦**：CM6 是 ESM-first，CDN 不如 CM5 顺手。坚持零构建可退用 CM5 或 CM6 预打包 UMD bundle。
2. **React Virtuoso 的 `followOutput` 在高频流式下会失效**（issue #317）——用 React 必须 batch 更新；HTMX 方案需自己写 ring buffer + 节流，不能裸接 SSE。
3. 所有 URL 均经搜索结果交叉确认，未编造；正式选型前建议浏览器实测（部分文档页 URL 可能随版本微调）。
